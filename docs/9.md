# Implementation Plan: M9 Orchestration API (Polling Architecture)

This plan outlines the steps to refactor the `POST /api/analyze` endpoint from a synchronous, blocking request to an asynchronous, polling-based architecture, as specified in the `system-design.md`.

## User Review Required
> [!IMPORTANT]
> This is a major architectural change. The frontend will need to be updated to handle this new flow. It will no longer receive the final result immediately from `POST /api/analyze`. It must poll `GET /api/analyze/{job_id}` until the `stage` is `"done"` or `"error"`.

## Proposed Changes

### 1. Update Data Models
We need to add models for the new API responses to `backend/models.py`.

#### [MODIFY] [models.py](file:///Users/mayanksharma/Downloads/New_Projects/think-before-share/backend/models.py)
Add the following classes:
- `AnalyzeResponse(BaseModel)`: For the immediate response to `POST /api/analyze` (contains `job_id`).
- `JobStatusResponse(BaseModel)`: For the response to `GET /api/analyze/{job_id}` (contains `job_id`, `stage`, `result`, `error`).

### 2. Refactor the Pipeline in `main.py`
We will rewrite `main.py` to support background tasks and an in-memory job store.

#### [MODIFY] [main.py](file:///Users/mayanksharma/Downloads/New_Projects/think-before-share/backend/main.py)
- **Job Store**: Add a global dictionary `jobs = {}` to act as an in-memory database for job statuses. (Note: For a production app, this would be Redis or a database, but in-memory is perfect for this prototype).
- **Background Pipeline**: Extract the massive pipeline logic currently inside `analyze_video` into a separate function: `async def run_analysis_pipeline(job_id: str, url: str)`.
  - Inside this function, we will update the job's `stage` at each step (e.g., `"resolving_video"`, `"fetching_transcript"`, `"extracting_claims"`, `"analyzing_evidence"`, `"generating_insights"`, `"done"`).
- **Update `POST /api/analyze`**: It will now generate a UUID for the `job_id`, add an entry to the `jobs` dictionary, spawn `run_analysis_pipeline` using FastAPI's `BackgroundTasks`, and immediately return `{ "job_id": <uuid> }`.
- **Add `GET /api/analyze/{job_id}`**: This new endpoint will look up the `job_id` in the `jobs` dictionary and return its current status. If the job is `done`, it will include the full `AnalysisResult`.
